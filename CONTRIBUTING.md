# Git規約・ガイドライン

## 基本運用
1. 編集を行う場合は、まず `issue` を作成し、内容を明確にすること。
2. 作成したら、 `develop` ブランチからブランチを切って行うこと。
3. 変更が完了したら、`develop` ブランチにプルリクエストを作成し、コードレビューを受けること。
4. レビューが完了し、承認されたら `develop` ブランチにマージすること。
5. 定期的に `develop` ブランチから `main` ブランチへリリースを行うこと。
6. バージョンはタグで管理すること。

## ブランチルール

### ブランチ命名規則
以下の2つのブランチは常時使用される主要なブランチである。
- main: メインブランチ
- develop: 開発ブランチ

新しい機能追加やバグ修正のためのブランチは、以下の命名規則に従うこと。
`<type>/#<issue-number>/<short-description>`

`type` は以下のいずれかを使用する。
- feature: 機能追加
- fix: バグ修正
- build: ビルド関連の変更
- ci: 継続的インテグレーション関連の変更
- docs: ドキュメント変更
- style: フォーマット変更（機能に影響しない変更）
- refactor: リファクタリング
- perf: 性能向上
- test: テストコード追加・修正
- chore: その他の雑務

`issue-number` は関連するIssueの番号を指し、3桁のゼロパディングを行う（例: 001, 023, 105）。

`short-description` は変更内容を簡潔に表す短い説明をケバブケースで記述する（例: add-login-feature, fix-header-bug）。

## コミットメッセージ規則
コミットメッセージは、変更した内容を日本語で説明する。
説明が簡潔にできない場合は、別のコミットに分けること。

### コミットメッセージフォーマット
Conventional Commits 1.0.0 を参考にした以下のフォーマットを使用する。
```
<type>(<scope>): <subject>

<detail>
```
`type` は以下のいずれかを使用する。
- feature: 機能追加
- fix: バグ修正
- build: ビルド関連の変更
- ci: 継続的インテグレーション関連の変更
- docs: ドキュメント変更
- style: フォーマット変更（機能に影響しない変更）
- refactor: リファクタリング
- perf: 性能向上
- test: テストコード追加・修正
- chore: その他の雑務

`scope` は変更の影響範囲を示す任意の文字列（例: auth, header, database）。

`subject` は日本語で記述する。

`detail` は変更内容の詳細説明を記述する（必要に応じて）。

### ベストプラクティス
- 頻繁にコミットを行うこと。
- 意味のある単位でコミットを行うこと。
- `main` ブランチや `develop` ブランチへの直接コミットしないこと。
